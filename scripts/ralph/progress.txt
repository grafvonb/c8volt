## Codebase Patterns
- Keep shared CLI failure classification in `c8volt/ferrors`; normalize raw errors with `Normalize` and assert `Classify` or `ResolveExitCode` directly in package tests before adding subprocess coverage in `cmd/`.
- Return CLI root/bootstrap failures through the shared `cmd` bootstrap helpers and `ferrors.HandleAndExit`; do not leave entry points on raw `os.Exit(1)` paths if the failure should participate in shared classification and `--no-err-codes`.
- For CLI command-validation helpers, attach the shared `ferrors` class when constructing the error if the same validation may surface through both Cobra `Execute()` and in-command `ferrors.HandleAndExit`; bootstrap normalization alone only covers the former.
- Extend command and bootstrap sentinel-to-class mappings through the shared rule tables in `cmd/cmd_errors.go` and `cmd/bootstrap_errors.go`; that table-driven path is the extension point for future CLI failure classes.
- In fresh helper-process `cmd` tests, skip the shared Cobra flag reset helper before `SetArgs()`; resetting `StringSlice` defaults can materialize a literal `"[]"` entry and send commands down the wrong path.
- Command handlers that call `NewCli(cmd)` must treat the returned config as nilable on failure; use the shared command/bootstrap error helper to resolve `--no-err-codes` instead of dereferencing `cfg.App.NoErrCodes` in the error branch.
- The scripting/error-code guidance in `README.md` and `docs/index.md` is duplicated on purpose; update both files together when the CLI failure contract changes.
- For feature closeout on this branch, the authoritative validation sequence is the spec quickstart order: `go test ./cmd/... ./c8volt/ferrors/... -count=1` first, then `make test`.
