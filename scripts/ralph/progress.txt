## Codebase Patterns
- Cluster subresources belong under `getClusterCmd`; keep each nested Cobra command in its own `cmd/get_cluster_<resource>.go` file and register it from `init()`.
- CLI tests that depend on version-specific API payloads should set `app.camunda_version` in the temp config explicitly instead of relying on the default test config version.
- CLI failure-path tests for Cobra commands should run `Root()` in a helper subprocess with `GO_WANT_HELPER_PROCESS=1` and a temp `C8VOLT_TEST_CONFIG` so `ferrors.HandleAndExit` exit codes can be asserted cleanly.
- Treat `docs/cli/` as generated output from Cobra command metadata and regenerate it with `make docs` after command-tree or help-text changes instead of hand-editing the Markdown pages.
- Versioned service packages mirror the shared facade by defining a local `API` interface in each `v87`/`v88` contract file, and factory tests should assert the concrete service type plus `services.ErrUnknownAPIVersion` for unsupported versions.
- When a generated-client service method uses the standard success path, prefer `internal/services/common.RequirePayload` for the shared HTTP-status plus non-nil JSON payload validation and keep version-specific exceptions, such as tolerant stats enrichment, explicit at the call site.
- For generated client endpoints that return XML or plain string payloads through `XML200`, use `internal/services/common.RequirePayload` as the empty-200 guard instead of duplicating string-specific malformed-response checks.
